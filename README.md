## شرح المصطلحات (لغير المختصين)

هذا الجدول يوضح معنى المصطلحات التقنية التي ترد في الوثيقة. يمكن الرجوع إليه عند القراءة.

| المصطلح | المعنى المبسط |
|---------|---------------|
| **قاعدة البيانات** | مكان تخزين منظم لبيانات المستخدمين والطلبات (مثل جدول إكسل ذكي). |
| **SQLite** | نوع بسيط من قواعد البيانات، يعمل كملف واحد ولا يحتاج خادماً منفصلاً. |
| **المطابقة الذكية** | آلية تقارن تخصص الطالب ومهاراته مع الجهة وتحدد مدى ملاءمتها (نسبة مئوية). |
| **التشابه الدلالي** | قرب المعنى بين نصين؛ مثلاً "هندسة البرمجيات" و"تطوير البرمجيات" متقاربان في المعنى. |
| **النموذج** | برنامج ذكاء اصطناعي مُدرَّب لفهم النصوص العربية وحساب التشابه بينها. |
| **embeddings** | تمثيل النص كأرقام (متجه) يفهمه الحاسوب؛ النصوص المتشابهة تكون أرقامها قريبة. |
| **Cosine similarity** | مقياس رياضيّ لمدى تقارب نصين؛ كلما زادت القيمة (حتى 1) زاد التشابه. |
| **عتبة** | حد أدنى؛ الجهات التي تقل ملاءمتها عن 40% لا تظهر للطالب. |
| **التدريب (Fine-tuning)** | تحسين النموذج بتمريره على بيانات عربية إضافية لزيادة دقته. |
| **API / مفاتيح API** | واجهة خارجية قد تتطلب اشتراكاً أو رسوماً؛ المنصة لا تعتمد عليها. |
| **Hugging Face** | منصة مجانية تحتوي على نماذج وبيانات جاهزة للذكاء الاصطناعي. |
| **arabic-sts** | مجموعة بيانات عربية تحتوي آلاف أزواج الجمل مع درجات تشابهها (0–5). |
| **GPU** | معالج رسوميات يساعد في تسريع التدريب (اختياري). |
| **fallback** | خطة بديلة؛ إذا فشلت المطابقة الذكية، تُستخدم قواعد ثابتة (كلمات مفتاحية). |
| **التحقق / الاختبار** | في التدريب: بيانات تُستخدم للتأكد من أن النموذج يعمل جيداً قبل الاستخدام الفعلي. |
| **loss (الخسارة)** | مقياس خطأ النموذج؛ كلما انخفضت، زادت دقة النموذج. |
| **admin / مدير النظام** | دور مستخدم يدير المشرفين والطلاب وإسناد المشرفين وعرض جميع المستخدمين. |
| **إسناد مشرف** | ربط الطالب بمشرف رسمي في قاعدة البيانات (حقل `assigned_supervisor_id`). |
| **CRN / شعبة** | الرقم المرجعي للمقرر ورمز الشعبة لربط الطالب بالمشرف والمقرر. |

## 1. شرح فكرة المشروع

منصة تعاون هي نظام ويب جامعي يربط بين ثلاثة أطراف رئيسية في عملية التدريب التعاوني: الطلاب الجامعيين، والشركات والجهات الحكومية المستضيفة للتدريب، والمشرفين الأكاديميين. تهدف المنصة إلى تنظيم عملية التنسيق بين هذه الأطراف وتسهيل إدارة طلبات التدريب ومتابعة حالاتها.

## 2. المشكلة التي يحلها المشروع

- **تشتت التنسيق**: عدم وجود قناة موحدة للتواصل بين الطلاب والجهات والمشرفين.
- **صعوبة المتابعة**: صعوبة تتبع حالة طلبات التدريب لكل طالب.
- **إدارة الملفات**: حاجة الطلاب لرفع السير الذاتية وتخزينها بشكل منظم.
- **الفصل بين الأدوار**: كل طرف يحتاج لوحة تحكم خاصة بمهامه (الطالب، الجهة، المشرف).
- **عدم ملاءمة التقديمات**: تقديم الطلاب لجهات غير مناسبة لتخصصاتهم.

## 3. الفئة المستهدفة

- **الطلاب**: طلاب الجامعة المسجلون في برنامج التدريب التعاوني.
- **الجهات**: المؤسسات الحكومية (التابعة للوزارات) والشركات الخاصة التي تستقبل متدربين.
- **المشرفون الأكاديميون**: أعضاء هيئة التدريس المشرفون على برامج التدريب.

## 4. التقنيات المستخدمة

| الطبقة | التقنية |
|--------|---------|
| الواجهة الأمامية | HTML, CSS, JavaScript (أدوات بناء صفحات الويب) |
| الواجهة الخلفية | Python, Flask (لغة برمجة وإطار عمل لتشغيل الخادم) |
| قاعدة البيانات | SQLite (ملف تخزين بسيط للبيانات) |
| المطابقة الذكية | sentence-transformers (أداة ذكاء اصطناعي لفهم النصوص وحساب التشابه) |
| الخط | IBM Plex Sans Arabic |

## 4.1 دقة نموذج المطابقة

| المقياس | القيمة التقريبية |
|---------|------------------|
| نموذج أساسي (قبل التدريب) | تشابه دلالي جيد على 50+ لغة منها العربية |
| بعد التدريب على arabic-sts | تحسن ملحوظ في دقة التشابه الدلالي للنصوص العربية |
| عتبة عرض الجهات | 40% فأعلى (الجهات أقل من 40 لا تظهر للطالب) |
| طريقة الحساب | قياس التشابه (cosine similarity) بين تمثيل النصوص (embeddings)، ثم تحويل النتيجة إلى نسبة 0–100 |

النموذج يدمج بين **المطابقة الدلالية** (تحويل النص لأرقام ومقارنتها) و**القواعد** (ربط التخصصات بكلمات مفتاحية للجهات). عند فشل المطابقة الذكية، تُستخدم القواعد كبديل (fallback).

## 4.2 أمثلة على المطابقة

### المدخلات

| الطالب | الجهة | النتيجة المتوقعة |
|--------|-------|------------------|
| تخصص: هندسة البرمجيات، مهارات: Python, SQL | وزارة الاتصالات، قسم تقنية المعلومات | نسبة عالية (70–95) |
| تخصص: محاسبة، مهارات: Excel | وزارة المالية، الشؤون المالية | نسبة عالية |
| تخصص: هندسة البرمجيات | وزارة الصحة، الموارد البشرية | نسبة منخفضة أو غير ظاهرة |
| تخصص: طب | وزارة الصحة، المستشفيات | نسبة عالية |

### مثال على المخرجات (لوحة الطالب)

```
الجهة: وزارة الاتصالات - قسم تقنية المعلومات
نسبة الملاءمة: 87%
السبب: مطابقة دلالية بناءً على تشابه المعنى بين التخصص والجهة
```

### مثال على بيانات التدريب (arabic-sts)

مجموعة arabic-sts تحتوي أزواج جمل عربية مع درجة تشابهها (من 0 إلى 5):

```
الجملة 1: الطالب يدرس هندسة البرمجيات
الجملة 2: التخصص في تطوير البرمجيات والحاسوب
درجة التشابه: 4.2 / 5
```

## 7. منطق عمل المشروع - كل جزئية

### 7.1 نظام المصادقة

**التسجيل**: تشفير كلمة المرور (تحويلها لنص غير قابل للقراءة) باستخدام آلية آمنة قبل حفظها.

```python
password_hash = generate_password_hash(password)
conn.execute('INSERT INTO users (...) VALUES (?, ?, ?, ...)', (name, email, password_hash, ...))
```

**تسجيل الدخول**: التحقق من البريد وكلمة المرور، ثم إنشاء جلسة (تذكر أن المستخدم مسجّل الدخول) تحتوي على رقم المستخدم ودوره.

```python
if user and check_password_hash(user['password'], password):
    session['user_id'] = user['id']
    session['role'] = user['role']
    return redirect(url_for('dashboard'))
```

**حالة الحساب**: إن وُجد عمود `account_status` في جدول `users` وكانت قيمته `suspended`، يُرفض تسجيل الدخول حتى يُعاد تفعيل الحساب من الإدارة.

**حماية الصفحات**: آلية تتحقق من أن المستخدم مسجّل الدخول ونوع حسابه (طالب/جهة/مشرف) قبل السماح بفتح الصفحة.

```python
def login_required(role=None):
    def decorator(f):
        @wraps(f)
        def decorated_function(*args, **kwargs):
            if 'user_id' not in session:
                flash('يجب تسجيل الدخول أولاً', 'error')
                return redirect(url_for('login'))
            if role and session.get('role') != role:
                flash('غير مصرح لك بالوصول لهذه الصفحة', 'error')
                return redirect(url_for('dashboard'))
            return f(*args, **kwargs)
        return decorated_function
    return decorator
```

### 7.4 مطابقة الطالب مع الجهات المناسبة

الدالة `match_student_to_companies_ai` تستخدم أداة **sentence-transformers** (نموذج ذكاء اصطناعي متعدد اللغات) لتحويل نصوص الطالب والجهة إلى تمثيل رقمي (embeddings) وحساب التشابه بينها. يعمل كاملاً على جهازك دون الحاجة لاشتراكات أو مفاتيح خارجية.

```python
# ai_matching.py - embeddings دلالية
model = SentenceTransformer('paraphrase-multilingual-MiniLM-L12-v2')
student_emb = model.encode([student_text])
comp_embs = model.encode(comp_texts)
sim = cosine_similarity(student_emb, comp_emb)  # تشابه دلالي
score = (sim + 1) * 50  # تحويل لـ 0-100
```

### 7.6 تحميل السيرة الذاتية - الصلاحيات

السيرة تصل للجهة فعلياً. لا يمكن تحميلها إلا من:
- **الطالب**: سيرته فقط.
- **الجهة**: سيرة طالب تقدّم لها فقط (يوجد طلب في applications).

```python
def download_cv(filename):
    if session.get('role') == 'student':
        if student['cv_file'] == filename:
            return send_from_directory(app.config['UPLOAD_FOLDER'], filename, as_attachment=True)
    elif session.get('role') == 'company':
        app_row = conn.execute(
            'SELECT a.id FROM applications a JOIN students s ON a.student_id = s.id '
            'WHERE s.cv_file = ? AND a.company_id = ?',
            (filename, company['id'])
        ).fetchone()
        if app_row:
            return send_from_directory(app.config['UPLOAD_FOLDER'], filename, as_attachment=True)
    flash('غير مصرح بتحميل هذا الملف', 'error')
    return redirect(url_for('dashboard'))
```

### 7.7 تدفق الطلبات (قبول/رفض)

1. الطالب يقدم طلباً لجهة مناسبة (مع السيرة).
2. يُدرج سجل في `applications` بحالة `pending`.
3. الجهة تعرض المتقدمين ويمكنها قبول أو رفض كل طلب.
4. عند القبول/الرفض يُحدَّث `status` في الجدول.

```python
@app.route('/application/<int:app_id>/accept', methods=['POST'])
@login_required(role='company')
def accept_application(app_id):

        'UPDATE applications SET status = ? WHERE id = ? AND company_id = ?',
```

2. **إنشاء الحساب**: اختيار نوع الحساب (طالب/شركة-جهة/مشرف) ثم تعبئة النموذج. الجهة الحكومية تختار الوزارة.
3. **تسجيل الدخول**: إدخال البريد وكلمة المرور، ثم التوجيه للوحة التحكم حسب الدور.
4. **الطالب**: يعرض بياناته، السيرة، المهارات، قائمة الجهات المناسبة فقط (مطابقة ذكية)، وحالة طلباته. يجب رفع السيرة قبل التقديم.
5. **الجهة**: تعرض المتقدمين (ذكور/إناث)، المقبولين، المرفوضين، مع إمكانية تحميل السيرة والقبول والرفض والتواصل.
6. **المشرف**: يعرض الجهات (مع الوزارة)، قوائم المشرفين حسب الجنس، **شُعب المقررات** (CRN + شعبة)، **طلاباً مرتبطين بالشعبة أو مسندين رسمياً**، وطلبات التدريب بحالاتها؛ ويمكنه **إضافة طالب** جديد يُربط بحسابه كمشرف.
7. **المدير (admin)**: لوحة إدارة للمشرفين (إضافة/تعديل/تعليق/حذف)، إدارة الطلاب (إضافة فردية، **إسناد طالب ↔ مشرف**)، وصفحة **جميع المستخدمين**؛ الدخول من البريد `admin@tawun.com`.

## 11. تحسين النموذج (التدريب الإضافي) باستخدام Hugging Face

النموذج الحالي مُدرَّب مسبقاً على لغات متعددة منها العربية. يمكن تحسين دقة المطابقة للنصوص العربية بتدريبه على مجموعة **arabic-sts** (آلاف أزواج الجمل العربية مع درجات تشابهها) من منصة Hugging Face المجانية.

### 11.1 تشغيل سكربت التدريب

```bash
pip install datasets sentence-transformers torch
python train_model.py
```

السكربت يحمّل مجموعة `MohamedRashad/arabic-sts` (أكثر من 11 ألف زوج جمل عربية مع درجات تشابه)، يدرّب النموذج، ويحفظه في مجلد `tawun-match-model`.

### 11.2 استخدام النموذج المُدرَّب

`ai_matching.py` يتحقق تلقائياً: إن وُجد مجلد `tawun-match-model` بجانب الملف، يستخدمه. وإلا يستخدم النموذج الأساسي.

### 11.3 تفاصيل مجموعة arabic-sts

| البند | القيمة |
|-------|--------|
| المصدر | MohamedRashad/arabic-sts على Hugging Face |
| عدد أزواج التدريب | ~11,600 زوج (جملة أولى + جملة ثانية + درجة التشابه) |

| المجال | نصوص عربية متنوعة (أخبار، ثقافة، علوم، إلخ) |

### 11.4 المتطلبات

| المورد | الوصف |
|--------|--------|
| بيانات | arabic-sts من Hugging Face (تُحمّل تلقائياً عند التشغيل) |

## 13. التحديثات والميزات الإضافية (كلية الاتصال والإعلام، الإدارة، الإسناد)

هذا القسم يُكمّل الوثيقة بدون إلغاء ما سبق: يصف الهوية الجامعية، الحقول الجديدة، والمسارات البرمجية ذات الصلة.

### 13.1 الهوية والواجهة

- اسم الكلية في الواجهة يُعرف من `COLLEGE_NAME` في `constants.py` (القيمة الحالية: **كلية الاتصال والإعلام**).

### 13.2 بيانات الطالب والمقرر والربط بالمشرف

### 13.3 دور المشرف: إضافة طالب

- المشرف يمكنه إنشاء حساب طالب جديد مرتبط به كمشرف مسند عبر `POST /supervisor/students/add` (النموذج في لوحة المشرف).

## 14. شرح الصفحات والخصائص بلغة بسيطة (للمبتدئين)

هذا القسم يشرح **ماذا ترى** و**ماذا يحدث** عند استخدام المنصة، دون الدخول في تفاصيل البرمجة. يمكنك تصوّر أن كل صفحة هي **شاشة** تعرض معلومات أو **نموذجاً** تملأه ثم تضغط «إرسال» فيُخزَّن في النظام ويُعرض لاحقاً في مكان مناسب.

### 14.1 القالب المشترك (الهيكل العام)

- معظم الصفحات تستخدم **هيكلاً واحداً** (قالب أساسي): شريط علوي، عنوان الكلية، وروابط حسب حالة المستخدم (مسجّل دخول أم لا).
- **إن لم تكن مسجّلاً**: غالباً ترى «تسجيل الدخول» و«إنشاء حساب».
- **إن كنت مسجّلاً**: تُوجَّه تلقائياً إلى **لوحة التحكم** المناسبة لدورك (طالب، جهة، مشرف، أو مدير).

### 14.2 الصفحة الرئيسية (`/`)

- **الغرض**: التعريف بالمنصة ودعوة الزائر لإنشاء حساب أو تسجيل الدخول.
- **كيف تعمل**: تعرض نصاً وروابط؛ لا تخزّن بيانات شخصية حتى تختار التسجيل أو الدخول.

### 14.3 تسجيل الدخول

- **الغرض**: التأكد أنك أنت صاحب الحساب (البريد + كلمة المرور).
- **كيف تعمل**: تُدخل البيانات؛ إذا صحيحة يُفتح لك **جلسة** (كأن النظام «يتذكرك» أثناء التصفح). إذا كان الحساب **معلّقاً** من الإدارة، يظهر تنبيه ولا يُسمح بالدخول حتى يُفعَّل الحساب.

### 14.4 إنشاء الحساب — ثلاثة مسارات

#### أ) تسجيل طالب

- **ماذا تملأ**: بيانات شخصية، **القسم والتخصص** من قوائم جاهزة، **معلومات المقرر** (الاسم، الرقم المرجعي CRN، الشعبة)، وغالباً رفع السيرة الذاتية لاحقاً من صفحة التعديل.
- **ماذا يحدث بعد الإرسال**: يُنشأ **مستخدم** في النظام + **سجل طالب** مرتبط به. هذه البيانات تُستخدم لاحقاً لربطك بمشرف يدرّس نفس الشعبة أو لإسنادك رسمياً.

#### ب) تسجيل جهة (شركة أو جهة حكومية)

- **ماذا تملأ**: نوع الجهة، التخصص/المجال المناسب للتدريب، وإن كانت حكومية تُختار **الوزارة** من قائمة.
- **ماذا يحدث**: يُخزَّن وصف الجهة ليُقارَن لاحقاً مع ملف الطالب عند **اقتراح الجهات المناسبة** أو عند تقديم الطلب.

#### ج) تسجيل مشرف تدريب

- **ماذا تملأ**: بياناتك الأكاديمية والقسم، مع تمييز **دكتور/دكتورة** لعرض الواجهة المناسبة.
- **ماذا يحدث**: يُنشأ حساب مشرف؛ من لوحته يضيف **شُعب المقررات** ويرى الطلاب المرتبطين.

### 14.5 لوحة الطالب

- **الغرض**: ملخص حالتك: بياناتك، حالة السيرة، **اسم الأستاذ/المشرف** المرتبط بمقررك أو بالإسناد، والجهات المقترحة.
- **كيف تُبنى الفكرة**: النظام يقرأ تخصصك ومهاراتك ويقارنها بوصف الجهات؛ النتيجة **قائمة جهات مناسبة** (مع نسبة تقريبية). لا تظهر جهات ضعيفة التطابق جداً حسب عتبة النظام.
- **التقديم على جهة**: يتطلب عادة أن تكون السيرة مرفوعة؛ ثم يُنشأ **طلب** يظهر عند الجهة بحالة «قيد المراجعة» حتى تقبل أو ترفض.

### 14.6 لوحة الجهة (الشركة / المؤسسة)

- **الغرض**: رؤية من تقدّم إليكم، وفرزهم (مثلاً حسب الجنس إن وُجد)، وقبول أو رفض الطلبات، وتحميل السيرة عند وجود طلب مشروع.
- **كيف تعمل**: الجهة لا ترى سيرة أي طالب إلا إذا كان هناك **طلب مرتبط** بينهما؛ هذا يحمي خصوصية الملف.

### 14.7 لوحة المشرف

- **الغرض**: إدارة الجانب الأكاديمي للتدريب التعاوني لمقرراتك.
- **شُعب المقررات**: تضيف من النموذج **اسم المقرر + CRN + الشعبة**. النظام يطابق الطلاب الذين سجّلوا **نفس الأرقام** فيملأ قائمة طلابك تلقائياً.
- **الإسناد الرسمي**: إن أسندك المدير لطالب معيّن، يظهر هذا الطالب ضمن اهتماماتك حتى لو اختلفت تفاصيل الشعبة لاحقاً (حسب إعدادات النظام).
- **إضافة طالب**: نموذج ينشئ حساباً طالبياً جديداً ويربطه بك كمشرف مسند — مفيد عند تسجيل طالب لم يُكمل التسجيل الذاتي.
- **باقي المحتوى**: عرض الجهات والوزارات للاطلاع، ومتابعة صورة عامة عن التدريب (حسب ما تُظهره الصفحة).

### 14.8 الملف الشخصي وتعديل البيانات

- **الصفحة العامة للملف**: تعرض ما هو مخزّن عنك (اسم، بريد، قسم، إلخ).
- **التعديل**: نموذج تغيّر فيه الحقول المسموح بها (مثل السيرة، المهارات، الصورة)؛ عند الحفظ تُستبدل البيانات القديمة في قاعدة البيانات وتنعكس في اللوحة فوراً بعد التحديث.

### 14.9 صفحة «من نحن» أو التعريف (`/about`)

- **الغرض**: شرح فكرة المشروع للزائر أو المستخدم؛ غالباً نص ثابت دون تخزين مدخلات.

### 14.10 لوحة المدير (الإدارة)

- **الفكرة العامة**: حساب **خاص بالإدارة** يرى النظام كاملاً من الأعلى.
- **المشرفون**: قائمة بجميع مشرفي التدريب؛ يمكن إضافة مشرف، تعديل بياناته، **تعليق** حسابه (يمنع الدخول)، أو حذفه مع معالجة آثار ذلك على الطلاب المسندين.
- **الطلاب**: قائمة بالطلاب، إضافة طالب يدوياً، و**اختيار المشرف المسند** لكل طالب من قائمة — هذا هو «الإسناد الإداري» الواضح للجميع.
- **جميع المستخدمين**: جدول يوضح من سجّل في النظام وأدوارهم؛ مفيد للمراجعة والدعم.

### 14.11 المطابقة الذكية (بكلمات بسيطة)

- **بدون تعقيد**: النظام يحوّل نص «تخصصك ومهاراتك» ونص «وصف الجهة» إلى أشكال رياضية يقارن بينها فيشبه «فهم المعنى» وليس مطابقة حرفية فقط.
- **إن تعثّرت المطابقة الذكية**: تُستخدم **قواعد بسيطة** (كلمات مفتاحية) كخطة بديلة حتى لا يبقى الطالب بلا اقتراحات.

### 14.12 ماذا يعني «بناء الصفحة» هنا؟

- لكل شاشة في المتصفح: **قالب HTML** يحدد العناوين والجداول والأزرار، و**ملف تنسيق** (`style.css`) يحدد الألوان والمسافات، و**منطق خلف الكواليس** يقرأ من قاعدة البيانات ويملأ الجداول بالقيم الصحيحة لكل مستخدم. المستخدم يرى النتيجة النهائية فقط؛ لا يحتاج معرفة البرمجة لاستخدام المنصة.
